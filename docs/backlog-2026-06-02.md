## 1. PDF ビューア — 注釈ツール

### 1.1 蛍光ペン（フリーハンド）— 実装済み・要検証

| 項目 | 内容 |
|------|------|
| 仕様 | 矩形ハイライトではなく、なぞった軌道だけ半透明イエローの線として PDF に焼き付け |
| API | `POST /api/highlight` に `type=marker` + `path_json`（正規化座標の点列） |
| 実装 | `backend/services/pdf_annotations.py` の `draw_freehand_marker`、`usePdfEditor` / `MainCanvas` |
| 課題 | 実機でストローク幅・色・保存後プレビューの一致をスモーク確認 |

### 1.2 消しゴム — 要修正（優先: 高）

| 項目 | 内容 |
|------|------|
| 現状 | UI・API 配線はあるが、**焼き付け済みの蛍光ペン線が消えにくい** |
| 原因 | 白い `draw_polyline` 上書きだけではレイヤーによっては残る。PDF 注釈の `delete_annot` だけでは焼き付け線は消えない |
| あるべき姿 | なぞった軌道に沿った **redaction（赤塗り除去）** + 交差する PDF 注釈の削除 |
| 実装案 | `pdf_annotations.py` に `erase_region` / redaction 版 `draw_freehand_eraser` を追加し、`main.py` の `type=eraser` から呼ぶ |
| フロント | プレビュー条件を `editPageImage` 限定ではなく `hasPreview` に（読み込み中でもなぞりプレビュー） |
| 操作 | マーカーと同様 **フリーハンドでなぞる**（矩形ドラッグはフォールバック可） |
| 参照 | `docs/api-contract.md` — `eraser` + `path_json` |

**完了条件:** 蛍光ペンで引いた線を消しゴムでなぞり、保存後 PDF とプレビュー両方で消えている。

## 2. マトリクス — 資料スロット（四角い箱）の編集 UI

### 2.1 iOS 風編集モード — 一部実装済み・要検証

| 項目 | 内容 |
|------|------|
| 仕様 | 長押し（または「枠を編集」）で枠がプルプル揺れ、ドラッグ並べ替え・名前変更・資料の外し |
| 実装 | `MatrixSlotGrid` / `SortableSlotCard`、`globals.css` の `slot-jiggle` |
| 永続化 | `localStorage`（`taxx-slot-layout:v1`）、キー `${clientId}:${periodKey}` |
| 課題 | 実機で長押し・D&D・完了ボタン・監査ボタンとの干渉を確認 |

### 2.2 反映スコープ（各社 vs 一括）— 未実装（優先: 中）

編集モード中に **枠テンプレ（並び・ラベル）の反映先** を選べるようにする。

| スコープ | 意味 | 反映先の目安 |
|----------|------|----------------|
| **この会社** | 表示中クライアントのみ | `currentClient` × 現在の `periodKey` |
| **担当分** | 担当スタッフの全クライアント | `effectiveStaff.clients` × 同じ期間 |
| **選択** | チェックした会社のみ | 担当分から複数選択 × 同じ期間 |
| **全社** | 閲覧可能な組織マスタ上の全クライアント | `orgClients` ∩ `canAccessClient` × 同じ期間 |

**原則**

- 一括で変えるのは **枠の並び・ラベル名**（テンプレート）のみ。
- **PDF の中身**は会社ごとに別キーのため、一括では触らない。「− で外す」は **この会社だけ**。
- 期間は一括でも **今見ている期間**（`perm` / `year:N` / `month:N`）に揃える。

**実装メモ（未着手）**

- `frontend/src/lib/slot-layout-scope.ts` — スコープ解決・一括 `persist`
- `SlotLayoutScopeBar.tsx` — 編集モード内のセグメント UI
- `page.tsx` — `layoutEditScope` / `selectedLayoutClientIds` / `applySlotLayout` の分岐

**議論が必要な点**

- 「全社」は **閲覧可能な全社** でよいか、マスタ全件か（権限 `document.layout.admin` 等の要否）。

**完了条件:** 担当分を選んで枠名を変え、別クライアント同期间でマトリクスを開くと同じラベル・並びになっている。

## 3. OCR と正規化データ — スコープの絞り込み（設計）

P3 本実装の前に **「OCR から何を正規化するか」** を固定する。詳細方針は本節および `docs/roadmap.md` P3、`docs/client-data-vision.md`（DATA 画面への自動反映）を参照。

### 3.1 用語の分離

| 用語 | 意味 | 例 |
|------|------|-----|
| **OCR** | 画像 PDF からテキストを取る／検索可能 PDF 化 | Tesseract、`/api/ocr` |
| **分類 v1** | テキスト＋ファイル名 → スロット候補 | `POST /api/classify`、`doc_classifier.py` |
| **Docugrid 正規化** | ページ順・ファイル ID・注釈 | Zustand `docugrid-store`（OCR 不要） |
| **証憑メタ正規化** | 業務で使う構造化フィールド | 将来 `metadata_json` / 正規化ストア |

**やらないこと:** OCR 生テキストを全部構造化して 1 スキーマに押し込む。

### 3.2 用途別ゴール

| 用途 | 正規化の最小セット | フェーズ |
|------|-------------------|----------|
| 自動振り分け | `suggested_slot` / `category_id` / `confidence` / `engine` / `text_excerpt` | **A（P3 v1）** |
| 要確認キュー | 上記 + `status: needs_review` | A |
| 不足資料 | カテゴリ + 期間 + 承認状態（OCR 以外も多い） | P4 |
| ダッシュボード・アラート | 税目・期限・金額（カテゴリ限定） | **C（P5 寄り）** |
| 全文検索 | 構造化ではなく検索可能 PDF またはインデックス | 別パイプライン |

### 3.3 書類カテゴリとの対応

`frontend/src/config/organization.ts` の `DocumentCategory` とフラグをそのまま採用する。

| フラグ | 意味 |
|--------|------|
| `ocrTarget` | OCR 正規化の対象候補 |
| `dashboardTarget` | ダッシュボード集計に使う |
| `alertTarget` | 届出アラート等 |

**フェーズ B:** `ocrTarget: true` のカテゴリ（申告書・元帳・通帳など）だけ **カテゴリ別フィールド allowlist** を定義する。

### 3.4 正規化 v1 スキーマ案（`ExtractedDocumentMeta`）

アップロード／分類時に `document_versions.metadata_json` へ格納する薄い層。

```json
{
  "client_id": "string (optional, upload context)",
  "period_key": "string (optional)",
  "suggested_slot_id": "string (optional)",
  "category_id": "DocumentCategory (optional)",
  "confidence": 0.0,
  "engine": "pymupdf | tesseract | openai | ...",
  "text_excerpt": "string (max ~400 chars)",
  "status": "processing | needs_review | linked | failed"
}
```

- 信頼度が閾値未満 → **正規化フィールドを増やさず** `needs_review` のまま。
- 人が確定した値と AI 推定は **別キー**（`confirmed_*` vs `suggested_*`）を推奨。

### 3.6 社内で決めるチェックリスト（30 分）

1. P3 v1 の成功指標は「スロット自動振り分け 80%」だけか、金額抽出は v2 か。
2. どの `DocumentCategory` に OCR 正規化をかけるか（`ocrTarget` でよいか）。
3. 人の上書きは正規化を上書きするか、別フィールドか。

5. 保存先は `metadata_json` のみか、集計テーブルが要るか。
