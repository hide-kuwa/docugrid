# 給与・源泉徴収簿・年末調整（まるふ）ビジョン

最終更新: 2026-06-17

## 位置づけ

**「まるふ（マル扶・マル基配・マル保）」の自動読み取り → 源泉徴収簿 → 社保・源泉の全自動計算 → 年末調整ワンクリック** は、年末調整という「日本最大級のペーパーワーク地獄」を根本から変える **聖杯** 機能として位置づける。

親文書: [`taxx-ecosystem-development-plan.md`](taxx-ecosystem-development-plan.md) §2.9、Phase 3  
UI 基盤: [`capture-gallery-ux-vision.md`](capture-gallery-ux-vision.md)（撮影・Masonry カード）

---

## 1. 実務のペイン（なぜ聖杯か）

| ペイン | 現状 |
|--------|------|
| 手書き年末調整 | 文字が読めない、配偶者所得の誤り、生命保険の新・旧区分ミス |
| 神経衰弱突合 | 申告書 ↔ 控除証明書 ↔ 給与ソフト再入力 ↔ 源泉徴収簿 |
| 給与ソフトの限界 | **きれいに入力されたデータ** を計算するだけ。アナログの山は人間が前処理 |
| 分断 | 給与計算・社保・源泉・年調・法定調書が別システム |

TAXX の価値は、「ノイズだらけの手書き書類・証明書（アナログ）」を読み取り、**1 円の狂いもない源泉徴収簿と計算ロジック（デジタル）** へ一本の線で繋ぐことにある。

## 2. 目標体験（To-Be）

### 2.1 まるふ × 控除証明書の OCR・RAG 自動検算

従業員はスマホで **マル扶・マル保・生命保険控除証明書** を撮影して DocuGrid にアップするだけで、作業の 9 割が完結する。

| 能力 | 説明 |
|------|------|
| **実態クロスチェック** | 申告書の手書き数字と控除証明書の証明額を自動突合。「正しい控除額はこちらです」と修正案を提示 |
| **要件判定** | マル基配の所得見積もりを前年・期中給与から算出。「配偶者特別控除の対象」等を自動判定し源泉徴収簿へマッピング |
| **グレー UX** | 判定が曖昧な場合はワンタップ「含む / 含まない」（`taxx-ecosystem-development-plan.md` §3 と同型） |

### 2.2 給与計算・社保・源泉の完全自律ループ

社会保険料（健保・厚年・雇保）と源泉所得税は **源泉徴収簿のデータ構造と直結** し、1 円のズレを許さない。

| 能力 | 説明 |
|------|------|
| **算定基礎届** | 4・5・6 月給与から自動集計 → 9 月からの標準報酬月額（等級）を源泉徴収簿に反映 |
| **社保自動切替** | 10 月支給から人間の設定なしで新保険料率を適用 |
| **税額表ルックアップ** | マル扶データに基づく甲欄・扶養人数の自動適用 |
| **期中扶養増減** | 源泉徴収簿リアルタイム更新 → 当月給与へ即反映 |

### 2.3 「年末調整を実行」ボタン

毎月の給与・賞与・社保・源泉が源泉徴収簿に蓄積され、OCR したまるふ控除が統合された状態で:

1. 担当者が **「年末調整を実行」** を押す  
2. 年調年税額・過不足額が一瞬で計算  
3. 12 月（または 1 月）給与に過不足を自動セット  
4. **源泉徴収票** と **法定調書合計表** が裏で完成

## 3. データ統合の北極星

以下が **TAXX 単一データベース** 内で完全統合されたとき、既存の会計ソフト・給与ソフトは「入力先」ではなく「出力先」に降格する。

```
固定資産 ─┐
支払調書 ─┤
給与台帳 ─┼─► 源泉徴収簿（SSOT）─► 仕訳・申告・法定調書
まるふ   ─┤
社保マスタ┘
```

関連: [`taxx-ecosystem-development-plan.md`](taxx-ecosystem-development-plan.md) §2.8 固定資産連動

## 4. データモデル（設計スケッチ・未実装）

### 4.1 源泉徴収簿（SSOT）

| エンティティ | 要点 |
|--------------|------|
| `Employee` | 従業員マスタ（扶養・配偶者・甲乙・社保等級） |
| `WithholdingLedger` | 月次給与・賞与・控除・社保・源泉税の行（1 円単位） |
| `YearEndAdjustmentRun` | 年調実行単位（スナップショット・過不足） |
| `DeductionCertificate` | 控除証明書 OCR 結果（保険・住宅等） |
| `MarufuSubmission` | マル扶 / マル基配 / マル保の構造化フィールド + 原本画像 ref |
| `SocialInsuranceGrade` | 標準報酬月額・等級の履歴（算定基礎届連動） |

### 4.2 書類種別と OCR スキーマ

| 書類 | 抽出フィールド例 |
|------|------------------|
| マル扶 | 扶養親族、障害、寡婦・ひとり親、所得見積 |
| マル基配 | 配偶者所得見積、控除区分 |
| マル保 | 保険料控除申告額 |
| 生命保険等控除証明書 | 証明額、新・旧区分、保険会社 |
| 源泉徴収票（参考） | 年間集計との突合 |

### 4.3 計算エンジン境界

| レイヤ | 責務 |
|--------|------|
| **ルール KB** | 税額表、社保料率、標準報酬等級表、通達（`taxx-ecosystem-development-plan.md` §2.6） |
| **Ledger Engine** | 月次給与 → 源泉徴収簿行の生成 |
| **Legal Master Client** | 社保料率・源泉税額表・控除額を `transaction_date` 指定で取得（[`temporal-master-pattern.md`](temporal-master-pattern.md)） |
| **Audit Engine** | 申告書 vs 証明書 vs 台帳の三角突合 |
| **YearEnd Engine** | 年調一括実行 → 過不足 → 法定調書 |

## 5. 実装フェーズ案

| 段階 | スコープ | 実務ゴール |
|------|----------|------------|
| **P-W1** | 源泉徴収簿データ構造 + 月次給与行の手動/CSV 取込 | SSOT の器 |
| **P-W2** | 控除証明書 OCR + 申告額 vs 証明額の突合 UI | 手書き検算の半自動化 |
| **P-W3** | マル扶系 OCR + 扶養・配偶者要件判定 | 年調入力の 9 割削減 |
| **P-W4** | 社保等級・算定基礎届連動 | 10 月切替の無人化 |
| **P-W5** | 年末調整実行 + 源泉徴収票・法定調書出力 | 「ポチるだけ」年調 |

現行リポジトリ: DATA タブの給与台帳スロット参照のみ。給与 SSOT は **未着手**。

## 6. 設計上の CHECK

| リスク | 対策 |
|--------|------|
| 手書き OCR 精度 | 証明書の機械読取額を優先。申告書はサジェスト + 人間確認 |
| 法改正・料率更新 | リーガルテック KB をバージョン管理。古い料率の警告 |
| 1 円ズレ | すべて整数円。台帳行と仕訳生成を同一トランザクション |
| 個人情報 | クライアント限定 RAG・テナント分離（`auth-tenancy-design.md`） |
